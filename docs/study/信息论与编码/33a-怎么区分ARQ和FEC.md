# 怎么区分ARQ和FEC

## 这类题在考什么

这类题通常考：

```text
发现错了是重发
还是自己纠
还是先纠不行再重发
```

也就是区分：

- `ARQ`
- `FEC`
- `HEC`

## 先记公式

这类题没有代数公式，  
最稳的是记三句话：

```text
ARQ：错了就重发
FEC：提前加冗余，接收端自己纠错
HEC：先纠错，纠不过再重发
```

## 做题步骤

### 第一步：先看接收端有没有“请求重发”

如果题目描述里有：

```text
请求发送端再发一次
```

那优先想到 `ARQ`。

### 第二步：看发送端是不是提前加了冗余

如果题目强调：

```text
发送时已经加入检错纠错位
接收端自己恢复
```

那就是 `FEC`。

### 第三步：看是不是两者结合

如果题目说：

```text
先尝试纠错
失败后再申请重发
```

那就是 `HEC`。

### 第四步：顺手判断适用场景

- 实时要求高、重发代价大，偏向 `FEC`
- 双向链路稳定、允许等待，偏向 `ARQ`

## 最简单例子

如果题目说：

```text
视频直播中，为了减少等待，不依赖重发，
发送端提前加入冗余让接收端自行恢复部分错误
```

那就是：

```text
FEC
```

如果题目说：

```text
数据包校验失败后，请求对方重新发送
```

那就是：

```text
ARQ
```

## 易错点

- ARQ 不是纠错，它主要靠重发。
- FEC 不是“发现错了再说”，而是提前加冗余。
- HEC 不是第四种新方法，本质上是两者结合。

## 注意点

- 这类题经常是选择题，关键词一抓就能秒。
- “能不能重发”“能不能等”是区分 ARQ 和 FEC 的关键场景线索。
